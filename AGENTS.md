# AGENTS.md

## 專案定位

這個專案用來陪使用者從零開始學習咖啡烘豆，並逐步累積可回顧、可比較、可調整的烘焙知識與烘焙計畫。

目前已知條件：

- 使用者是烘豆初學者，預設沒有烘豆背景知識。
- 烘豆機是新款、非 Lite 的 Kaleido M1 Artisan 版，工作電壓與整機額定總功率為 `110V / 1000W`；使用者拆檢確認共有兩支加熱管，每支額定 `400W`，加熱元件額定功率合計 `800W`。其餘約 `200W` 額度供風扇、滾筒馬達、照明與控制電路等使用，各部件實際分配、加熱管接線與控制板驅動方式仍待確認。機器容量以 200g 等級理解；實際批次量以每支豆子的計畫為準，目前可用 `100g` 左右小批次建立基準。
- 烘焙紀錄與曲線觀察使用 Artisan。
- 規劃或覆盤 Kaleido M1 前，先以 `docs/equipment/kaleido-m1-machine-and-heat-system.md` 作為機器結構、熱能來源與控制邏輯基準，再搭配 `docs/equipment/kaleido-m1-sv-burner-air-relationship.md` 的本機實測；報告列為「待確認」的項目不能自行假設。
- 專案內容以文件為主，用來科普烘豆知識、整理烘焙方法、保存烘焙計畫與回顧。

## 回答與文件語言

- 使用繁體中文，語氣清楚、務實、適合初學者。
- 預設用台灣常見咖啡與烘豆用語。
- 專有名詞第一次出現時，先用白話解釋，再附英文或縮寫。
- 不要假設使用者知道烘豆術語，例如 BT、ET、ROR、FC、DTR、charge、turning point。
- 避免只給抽象概念；盡量搭配「為什麼重要」、「怎麼觀察」、「下一次怎麼調」。

## 專案目錄規劃

目前優先依「豆子」分資料夾。每一支豆子有自己的生豆資料、參考曲線、每一鍋計畫、Artisan 檔案與覆盤，方便同一支豆跨鍋比較。

```text
docs/
  knowledge/        # 烘豆基礎知識、名詞解釋、觀念科普
  equipment/        # Kaleido M1、Artisan、感測與操作設定
    tests/           # Kaleido M1 空機或設備控制測試的 Artisan 原始資料
  tasting/          # 杯測、風味描述、萃取與烘焙結果關聯

beans/
  <bean-slug>/       # 單一豆子的總資料夾，例如 ethiopia-nensebo-geisha-washed
    bean.md          # 生豆基本資料、目前學到的重點、檔案索引
    references/      # 這支豆或這支豆使用到的參考曲線、截圖、供應商資料
    roasts/
      YYYY-MM-DD-1st-roast/
        plan.md      # 該鍋原始計畫
        review.md    # 該鍋覆盤；若尚未烘可先沒有
        artisan/
          *.alog     # Artisan 原始檔
          *.csv      # Artisan 匯出資料
          *.png      # 曲線圖或截圖
      YYYY-MM-DD-2nd-roast/
        plan.md
        review.md
        artisan/

plans/
  templates/         # 跨豆子可重複使用的烘焙計畫模板
  upcoming/          # 臨時草稿；正式整理後移到 beans/<bean-slug>/roasts/
  completed/         # 舊資料或尚未整理成豆子資料夾的已完成計畫
```

豆子資料夾命名建議用小寫英文與連字號，例如 `ethiopia-nensebo-geisha-washed`。若同一支豆有不同批次或不同供應商，檔名或資料夾名稱要補上足以區分的資訊。

烘焙鍋次資料夾與該鍋 Artisan 檔案命名使用數字序數，不使用英文單字序數，避免第很多鍋時難讀。格式是 `YYYY-MM-DD-<ordinal>-roast`，例如 `2026-07-01-1st-roast`、`2026-07-02-2nd-roast`、`2026-07-03-3rd-roast`、`2026-09-10-99th-roast`。序數後綴照英文慣例：`1st`、`2nd`、`3rd`，其他多數用 `th`，但 `11th`、`12th`、`13th` 仍用 `th`。

Kaleido M1 空機或設備控制測試放在 `docs/equipment/tests/<YYYY-MM-DD>-<test-topic>/`。資料夾與原始檔使用相同的日期式主名稱，例如：

```text
docs/equipment/tests/2026-07-08-sv-burner-air-empty-drum/
  2026-07-08-sv-burner-air-empty-drum.alog
  2026-07-08-sv-burner-air-empty-drum.csv
  2026-07-08-sv-burner-air-empty-drum.png
```

設備測試至少保留 `.alog`、`.csv`、`.png` 與文字結論。搬移或改名時要確認原始檔內容沒有變更，不要只留下整理後的摘要。

## 文件寫作原則

- 文件應該能讓一個剛開始烘豆的人讀懂。
- 每篇知識文件建議包含：
  - 主題是什麼
  - 為什麼對烘豆重要
  - 初學者要觀察什麼
  - 常見誤解或錯誤
  - 跟 Kaleido M1 / Artisan 實作上的關聯
- 若資訊涉及特定機器參數、Artisan 設定、韌體或軟體版本，不要憑記憶亂寫；先確認來源或在文件中標註「待確認」。
- 可以使用表格整理比較，但不要為了格式犧牲可讀性。
- 文件檔名建議使用小寫英文與連字號，例如 `rate-of-rise.md`、`first-crack.md`。

## 烘焙計畫與覆盤原則

烘焙計畫應該能回答：

- 這次烘什麼豆子？
- 想達成什麼風味或烘焙程度？
- 預計下豆量是多少？
- 預計入豆溫、火力、風門或鼓風設定是什麼？
- 預計在哪些時間點觀察或調整？
- 預計一爆、發展時間、下豆溫或下豆條件是什麼？
- 實際結果跟預期差在哪裡？
- 下一鍋要改什麼？

每份烘焙計畫優先放在：

```text
beans/<bean-slug>/roasts/YYYY-MM-DD-<ordinal>-roast/plan.md
```

每份烘焙計畫至少要有：

```markdown
# 烘焙計畫：豆名 / 日期

## 目標

## 生豆資料

## 設備與批次

## 預計曲線

## 操作步驟

## 實際紀錄

## 杯測與回顧

## 下次調整
```

每份覆盤優先放在：

```text
beans/<bean-slug>/roasts/YYYY-MM-DD-<ordinal>-roast/review.md
```

覆盤至少要有：

```markdown
# 第 N 鍋覆盤：豆名 / 日期

## 檔案

## 基本資料

## 曲線事件

## 當下觀察

## 主要判讀

## 第二鍋 / 下一鍋要驗證什麼
```

覆盤要先整理事實，再分析可能原因，最後提出下一鍋可驗證的調整。不要把單一結果寫成絕對規則。

## Artisan 相關紀錄

- 優先保留原始曲線或匯出資料，不要只留下文字心得。
- 新資料優先放在對應豆子的 `artisan/` 資料夾，例如 `beans/<bean-slug>/roasts/YYYY-MM-DD-1st-roast/artisan/`。
- 文字筆記要對應曲線事件，例如入豆、回溫點、一爆開始、一爆結束、下豆。
- 若引用 Artisan 指標，先寫清楚數值來源與定義，避免把不同設定下的數值直接比較。
- 若 Artisan 的 `Roast -> Properties` 沒有填實際生豆重量與熟豆重量，文件中要明確標註，不要把圖上顯示的批次重量當作實際入豆量。
- 若有 reference 曲線或截圖，放在 `beans/<bean-slug>/references/`，並在計畫或覆盤中連結。

## 安全與實作注意

- 烘豆會產生高溫、煙霧與銀皮，文件中涉及操作時要提醒通風、防火與設備看顧。
- 不要建議使用者離開正在運轉的烘豆機。
- 規劃 Kaleido M1 烘焙時，操作前要確認集銀皮盒與排氣路徑；日曬、蜜處理、特殊發酵等高銀皮豆或連續烘焙要另外寫出銀皮監看方式。
- 說明銀皮起火時，要同時整理原廠規則與社群實務，清楚標示證據層級、適用情境與風險。不能因做法來自社群就省略，也不能把玩家做法寫成原廠保證的程序。
- 原廠尚未確認本機集銀皮盒起火時的 Air、Drum 與斷電順序。不得自行補成已知程序；應引用機器基準文件的玩家案例、消防底線與待確認清單。
- 不要把單一曲線或單次結果當作絕對標準；烘豆調整要以多次紀錄、杯測與一致條件比較。

## Kaleido M1 前置閱讀與空機測試規則

- 任何 Kaleido M1 烘焙計畫、下一鍋調整、Artisan 曲線覆盤、熱量判讀或操作建議，都必須先完整閱讀：
  - `docs/equipment/kaleido-m1-machine-and-heat-system.md`
  - `docs/equipment/kaleido-m1-sv-burner-air-relationship.md`
- 機器基準文件負責本機世代、控制版本、工作電壓、額定功率、加熱管、熱傳與待確認資訊；實測文件負責本機 SV、Burner、Air 的空機控制行為。兩份文件要一起使用。
- 文件列為「待確認」、兩份資料互相衝突，或缺少的實機資訊會改變熱能判讀時，先向使用者提問。不能用其他 M1 世代、Lite、Dual、舊版手冊、系列商品頁或 Artisan 預設值補成這台機器的事實。
- 2026-07-05 與 2026-07-08 的 SV、Burner、Air 測試都是空機測試，沒有實際入豆。
- 2026-07-08 測試按下 CHARGE，只為了讓 Artisan 進入正式記錄流程。`.alog` 中殘留的豆名與生豆重量 `101g`，以及為記錄流程按下的 CHARGE、DROP 和相關 roast phase 衍生值，都不能當成實際烘焙或入豆證據。
- 空機測試只能用來理解控制與探針反應；正式烘焙建議仍要用實際豆量、曲線事件、熟豆重量、杯測與多鍋一致條件驗證。

## 協作方式

- 當使用者要求「科普」時，先用基礎概念說明，再補上與實際烘焙計畫的連結。
- 涉及 Kaleido M1 的機器原理、功率、控制邏輯、烘焙計畫或覆盤前，依「Kaleido M1 前置閱讀與空機測試規則」先讀兩份基準文件，再確認這台實機的世代、控制版本、工作電壓、加熱管支數與單支功率。若資料缺漏、來源互相衝突，或答案會改變熱能判讀，必須先向使用者提問。
- 當使用者要求「規劃一鍋」時，先確認豆子資訊、目標風味、批次量與目前可控制參數；資訊不足時，可以先做保守假設並標註。
- 當使用者貼 Artisan 曲線或烘焙紀錄時，先整理事實，再分析可能原因，最後提出下一鍋可驗證的調整。
- 不要一次給太多進階理論；以「下一次烘豆能用上」為優先。
